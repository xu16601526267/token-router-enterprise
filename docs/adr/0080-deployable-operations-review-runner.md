# ADR 0080: Deployable operations review runner

## Status

Accepted

## Context

The product principles define the current AI operations boundary as "agent produces insights and recommendations, humans approve in the dashboard." The architecture already has many agent-readable review surfaces:

- `SupplierScorecard`
- `SupplierPostureRecommendation`
- `TrafficProfile`
- `TrafficForecast`
- `PricingRecommendation`
- `SupplyDecision`
- `SupplyExpansionOpportunity`
- `OperatingInsight`

These surfaces are individually generated by admin APIs and proven in the real-process simulator. The remaining operational gap is deployment shape: telemetry has a deployable `token-router-supply telemetry sweep|agent`, but the broader operations review chain still depends on an operator clicking each generate button in the dashboard.

Without a deployable runner, P5/P7 remains less concrete than the data model suggests: the system can store recommendations, but there is no simple cron/systemd entrypoint that makes the agent-readable review set current.

## Decision

Add `token-router-supply review once`.

The command will call existing admin generate APIs in dependency order:

1. `/api/supplier_scorecards/generate`
2. `/api/supplier_posture_recommendations/generate`
3. `/api/traffic_profiles/generate`
4. `/api/traffic_forecasts/generate`
5. `/api/pricing_recommendations/generate`
6. `/api/supply_decisions/generate`
7. `/api/supply_expansion_opportunities/generate`
8. `/api/operating_insights/generate`

It will print a JSON summary with each step's returned count and `total_generated`. Operators can gate smoke runs with `--min-generated`.

The runner accepts explicit period/model/supplier filters and defaults to the same admin API base/token pattern as the telemetry runner. It does not add a new table in this ADR; the existing generated rows are the durable source of truth.

## Boundaries

`review once` must not:

- approve, reject, acknowledge, dismiss, apply, activate, disable, complete, or cancel anything
- modify `Channel.weight`, `Ability.weight`, `SupplyRoutingPolicy`, pricing tables, billing, settlement, wallet, payout, invoice, or funds state
- create action plans from unapproved decisions
- create suppliers, channels, capacity snapshots, cost profiles, prepaid lots, or executions

It only refreshes review read models through existing generate APIs.

## Consequences

- P5 becomes deployable without weakening the human-approval boundary.
- The same gb10 process run can prove the review runner calls the full evidence generation chain.
- The runner can be scheduled later by cron/systemd or a resident agent loop, but this ADR only adds `once` mode.
