# ADR 0001: 以 QuantumNous/new-api 作为 token-router 代码基线

- 日期：2026-06-22
- 状态：Accepted
- 关联原则：P2 无数据不承诺、P3 先度量再承诺 SLA、P8 守住边界、P9 流量即情报
- 关联架构：M0 数据骨架、M1 计量闭环、M2 会话亲和路由

## 背景

当前仓库只有产品原则与架构文档，还没有可运行代码。架构文档已明确本项目不是从零实现一个轻量代理，而是基于公开 `QuantumNous/new-api` 的 Go 单体，复用它已经存在的渠道、令牌、用户、分组、模型倍率、调用日志与 quota 统计能力。

token-router 的新增价值不在支付或钱包，而在：

1. 会话亲和路由，让同一 `SessionId` 尽量落到同一上游缓存池。
2. cache-aware 双价台账，让每次调用都能复算成本、收入与毛利。
3. 对账与流量画像，让线下财务、供应商评级与后续供给三轨有可信数据。

如果先自研一个小网关，会重复实现 new-api 已有的 OpenAI 兼容入口、渠道治理、token 配额、倍率与日志能力，并且更容易偏离架构文档的真实挂载点。

## 决策

把公开 `QuantumNous/new-api` 导入本仓库作为代码基线，保留本仓库现有 `README.md` 与 `docs/` 作为 token-router 的路线源。

导入原则：

1. 第一轮只做机械导入，不混入供应商、台账、亲和路由等业务改造。
2. 后续每个业务增量先写 ADR，再施工，再用实测验证。
3. 不引入支付网关、真实充值扣费、打款执行、发票/收款等资金动作。
4. 供应侧起步按第三方供给处理；流量供给环境先以 `gb10-4t` 作为简单供给目标，部署测试可先落在 `aima2`，后续迁移到专属服务器。

## 影响

正向影响：

- 立即获得可运行的 Go 单体、Docker 编排、OpenAI 兼容 API、渠道与令牌底座。
- M0/M1 的改造可以落在架构文档列出的真实文件与函数上，而不是临时代理。
- 后续迁移到专属服务器时，运行形态仍是标准 new-api 风格服务，迁移面较小。

代价：

- 初始提交会引入较多上游文件。
- 后续升级上游需要区分机械同步与 token-router 自研改动。
- 本仓库的 `README.md` 与上游 README 不能简单覆盖；token-router 文档必须继续作为本项目决策源。

## 验证

本 ADR 对应的施工完成后，需要至少证明：

1. 仓库存在 new-api 基线代码与依赖清单。
2. `go test ./...` 或当前环境下可执行的最接近编译验证能运行，并记录失败原因。
3. 本项目的 `README.md` 与 `docs/` 没有被上游导入覆盖。
4. `git status` 能清楚区分 ADR 与机械导入文件。
