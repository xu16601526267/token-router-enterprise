# token-router

> 双边 Token API 中转站（数据支撑型）—— Approaching.AI TaaS 系列

一边接入企业提供的 token 流量（上游/供给侧），一边把统一的 OpenAI 兼容 API 卖给其他企业（下游/需求侧），中间做精准计量与对账数据支撑。

## 一句话定位

**这是一个"会话亲和路由 + cache-aware 计量 + 对账数据导出"的系统，不是一个支付/钱包系统。**

## 两条关键定性

1. **平台不发生真实资金流。** 上、下游结算都走**线下公司级财务**；线上平台只做**数据支撑**——精准计量 + 产出成本/收入/毛利的对账数据。→ 不做支付网关、充值扣费、打款、发票、收款。
2. **成本的核心杠杆是 KV cache 命中率，不是 token×单价。** 因此 **session ID 的记录与分发**是平台核心：同一会话"亲和路由"到同一上游节点才能复用 KV cache，成本核算必须 **cache-aware**。对应 Mooncake / SGLang / KTransformers 的 KVCache 中心化后端。

## 产品理念（京东严选定位）

**做 token 领域的"京东严选"——不做最便宜的 token 集市，做企业级品质与稳定性最值得信赖的 token 严选中转站。**

严入选、SLA 分层、供应商评级、AI agent 驱动经营，目标是低运营成本下让供需两端都得到更满意的质量与价格，数据可信、质量可控。三个支点让"品质优先 + 低成本 + 两端共赢"自洽：**KV cache**（真实效率→正和共赢）、**可信数据**（平台唯一真资产）、**AI agent**（让高品质与低成本并存）。

供给侧分**三轨**（由轻到重）：第三方代销 → 自营垫资自采 → 自持算力，往哪轨扩由**流量数据**驱动（记录供需两端流量 → 分析 → 供给决策，agent 出洞察、人在 dashboard 批准）。

完整理念与九条原则见 [`docs/product-principles.md`](docs/product-principles.md)；流量记录、分析与供给延展设计见 [`docs/traffic-and-supply.md`](docs/traffic-and-supply.md)。

## 技术路线

- **基线**：公开 [`QuantumNous/new-api`](https://github.com/QuantumNous/new-api)（Go 单体：路由 + 计量 + 数据）。
- 对 new-api 主链路侵入极小（`Channel` 加一个字段 + 三处各插一行 + session 透传一处），~80% 工作量是旁边新增的独立模块：**成本台账 + 会话亲和路由 + 报表**。
- 从 `kvcache-ai/new-api` 移植 `service/channel_affinity.go`（会话亲和路由）。
- Higress 作为未来可选数据面，本期不做。
